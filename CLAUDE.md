```bash

docker compose -f docker-compose.dev.yml up -d

```

### Required Environment Variables
The application requires these env vars (typically in `.env` or passed directly):
- `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USERNAME`, `DB_PASSWORD` — MySQL connection
- `REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD` — Redis connection
- `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` — JWT signing secrets
- `COOKIE_DOMAIN`, `COOKIE_SECURE`, `COOKIE_SAME_SITE` — Cookie config
- `APP_CORS_ALLOWED_ORIGINS` — CORS origin whitelist

Swagger UI is available at `/swagger-ui/index.html` when running locally.

## Architecture Overview

**Stack**: Spring Boot 4.0, Java 21, Spring Security (JWT, stateless), Spring Data JPA, Flyway, Redis, AWS S3, MySQL

### Package Structure

```
ssafy.study.backend
├── domain/
│   ├── auth/         — Login, token reissue, logout; refresh tokens stored in Redis
│   ├── member/       — Signup, profile management
│   └── edu/
│       ├── study/    — Study groups (top-level; BACKEND/FRONTEND/ALGORITHM types)
│       ├── curriculum/ — Curricula within a Study
│       ├── post/     — Posts within a Curriculum (markdown content)
│       ├── image/    — S3 image management (presigned URL flow)
│       └── notice/   — Announcements
└── global/
    ├── response/     — ApiResponse<T> wrapper for all responses
    ├── exception/    — CustomException, ErrorCode enum, GlobalExceptionHandler
    ├── security/     — JWT filter, SecurityConfig, CookieService
    └── aws/          — S3Service (presigned URL generation/deletion)
```

### Key Patterns

**Response format**: All endpoints return `ApiResponse<T> { message, data }`. Use `ApiResponse.success(message, data)` for successes and `ApiResponse.error(ErrorCode)` for errors. Errors are thrown as `CustomException(ErrorCode)` and handled centrally by `GlobalExceptionHandler`.

**Authentication**: JWT tokens are passed and returned via **HttpOnly cookies** (`access-token`, `refresh-token`). `JwtAuthenticationFilter` extracts and validates the access token per request. Refresh tokens are stored in Redis (keyed by memberId) and support rotation on reissue.

**Authorization**: `SecurityConfig` permits only auth and signup endpoints publicly. Controller-level method security (`@PreAuthorize`) is enabled via `@EnableMethodSecurity`. Admin-only endpoints use `ROLE_ADMIN`.

**Database migrations**: Schema is managed exclusively by **Flyway** (`src/main/resources/db/migration/`). `ddl-auto` is set to `validate` — never use `create` or `update`. Add new migrations as `V{n}__description.sql`.

**Entity conventions**: Entities use Lombok `@Builder` with `id` excluded from the builder (auto-generated by DB). `@NoArgsConstructor(access = PROTECTED)` is required. `@Id` fields are never set directly.

**Image upload flow**: Client requests a presigned upload URL via `POST /posts/{postId}/images/presigned-url` → uploads directly to S3 → calls `PATCH /api/v1/images/{imageId}/complete` to confirm. Download uses a separate presigned GET URL with 10-minute expiry.

### Domain Hierarchy
```
Study (type: BACKEND/FRONTEND/ALGORITHM, level: BASIC/INTERMEDIATE/ADVANCED)
  └── Curriculum (ordered within Study)
        └── Post (ordered within Curriculum, markdown content)
              └── Image (S3-backed, presigned URL)
```

### Testing Conventions
Follow the **Fixture pattern** documented in `docs/test/jpa_test_guide.md`:
- Test fixtures live in `src/test` as `{Entity}Fixture` classes
- Use `ReflectionTestUtils.setField(entity, "id", value)` to inject IDs in unit tests — never add test helpers to production entities
- Expose `{Entity}.{Entity}Builder` from fixtures to allow field customization
